TÜRKÇE SÖZLÜK LAB - KATKI REHBERİ

Bu proje, Node.js, Express ve MySQL altyapısını kullanarak oluşturulmuş bir Türkçe sözlük web uygulamasıdır. TÜRKÇE SÖZLÜK LAB – ÖZEL LİSANS kapsamında yayımlanmaktadır. Ticari kullanım ve yeniden dağıtım, proje sahibi(ler)inin yazılı iznini gerektirir. Yine de projeye çeşitli biçimlerde katkı yapabilir, hata bildirebilir veya yeni özellikler önerebilirsiniz. Bu doküman, katkı sürecinde izlemeniz gereken adımları ve uyulması beklenen kod standartlarını açıklar.

GENEL BİLGİLER VE AMAÇ

Projenin Amacı:

  • Bu proje, kullanıcıların Türkçe kelime tanımları, eş anlamlılar, zıt anlamlılar, deyimler, alıntılar gibi sözlük öğelerine erişebildiği, Node.js & Express tabanlı bir web uygulamasıdır.
  • Aynı zamanda Admin Paneli üzerinden kelime ve anlam (definition) yönetimi sağlar.
  • Proje şu anda öğretici/demonstrasyon amaçlı kullanım önceliklidir; ticari kullanım proje sahibinden yazılı izin almadan yasaktır.

Katkı Profili:

  • Projeye ister yeni özellik ekleyebilir (örn. "kelime silme" endpoint'ini hayata geçirmek), ister hata tespiti/hata düzeltme, isterse dokümantasyon geliştirme olsun, katkıda bulunabilirsiniz.
  • Kod eklerken veya değiştirirken lütfen bu rehberdeki kurallara uyun.

KURULUM VE GELİŞTİRME ORTAMI

  1. Depoyu Klonlama ve Bağımlılıklar: 
    git clone https://github.com//.git
cd proje_adi
npm install
  2. .env Dosyası Düzenleme:

    Proje .env üzerinden veritabanı bilgilerini (DB_HOST, DB_USER, DB_PASSWORD, DB_NAME) ve SESSION_SECRET, ADMIN_PASSWORD, PORT gibi değerleri kullanır. Örnek .env dosyasını düzenleyip kaydedin.

  3. Veritabanı Oluşturma:

    sozluk_db adlı veritabanını oluşturun ve words, definitions, relations tablolarını çalıştırın (SQL şema veya migration varsa uygulayın).

  4. Geliştirme Sunucusu: 
    npm start

    Ardından http://localhost:3000 adresine giderek uygulamayı test edebilirsiniz.

    Daha fazla detay için projenin README.md dosyasına bakın.

KATKI SÜRECİ

Issue Açma

  • Hata veya Öneri: Bir hata bulduysanız veya yeni bir özellik fikriniz varsa, önce Issue’lar kısmına bakın. Benzer bir konu açıksa oradan devam edin, yoksa yeni bir issue açın.
  • Netlik ve Detay: Issue açıklamasında durumu net anlatın: nasıl tekrarlanır, hangi sürümde meydana gelir, hangi tarayıcı/ortamda. Yeni özellik talebinde bulunuyorsanız amacınızı/önerinizi detaylandırın.
  • Etiket ve Atama: Uygun label (ör. bug, enhancement) seçin veya proje sahibi ekleyecektir.

Pull Request Süreci

  1. Fork ve Branch Oluşturma: Depoyu fork edip kendi hesabınıza çekin, sonrasında feature/..., fix/... benzeri bir branch açın.
  2. Değişiklik Yapma: Kodunuzu yazarken Kod Standartları bölümüne dikkat edin. Hata düzeltme veya yeni özellik eklerken test etmeyi unutmayın.
  3. Test ve Kontroller (Varsa): Testleri (npm test gibi) çalıştırın, lint/format araçlarını (Prettier, ESLint) kullanıyorsanız hatasız olduğundan emin olun.
  4. PR Açma: Kendi branch’inizden projenin main veya develop dalına (projede hangisi esas alınıyorsa) karşı bir Pull Request açın. Açıklamada hangi sorunu çözdüğünüzü veya hangi özelliği eklediğinizi belirtin.
    Not: “Kelime silme” özelliği şu an için taslak olarak tanımlı; siz bu özelliği geliştirip PR açmak isterseniz memnuniyetle değerlendirilecektir.
  5. İnceleme ve Geri Bildirim: Proje sahibi veya diğer katkıcılar yorum yapabilir, düzeltme isteyebilir. Tartışmayı yapıcı biçimde sürdürerek gerekiyorsa ek commit’lerle PR’ı güncelleyin.
  6. Onay ve Birleştirme: PR onaylandığında proje sahibi dalınızı develop veya main dalıyla birleştirir. Tebrikler, katkınız proje geçmişine eklenir! 🤗

Kod İncelemesi ve Onay

  • Değişikliklerin proje ile uyumlu olması, testleri geçmesi, dokümantasyonda çelişki yaratmaması önemlidir.
  • Geri bildirimleri sabırla değerlendirip gerekiyorsa değişiklik yapın. Tüm testler ve incelemeler tamamlanınca PR birleştirilir.

Katkı Türleri

  • Yeni Özellik (feat): Henüz uygulamada olmayan ek fonksiyonlar (ör. “Kelime Silme Endpoint’i eklemek”).
  • Hata Düzeltme (fix): Mevcut koddaki bug’ları gidermek.
  • Dokümantasyon (docs): README, CONTRIBUTING veya inline yorumlarda düzeltme/geliştirme.
  • Refaktör (refactor): Kod kalitesini artırma, performans iyileştirme, vs.
  • Test (test): Yeni test ekleme veya mevcut testleri iyileştirme.

KOD STANDARTLARI

Proje Node.js (>=14) ve Express kullanıyor. Temel prensipler:

Node.js ve Express Kodlama Kuralları

  1. Dil ve İsimlendirme: Değişken, fonksiyon, dosya isimleri İngilizce ve camelCase (dosyalar kebab-case) olsun. Kod içinde Türkçe metin gerektiren yerlerde (ör. hata mesajı) elbette Türkçe kullanılabilir.
  2. Asenkron Yapı: Callback yerine mümkün olduğunca async/await tercih edin.
  3. Hata Yönetimi: Hataları try/catch veya Promise.prototype.catch ile yakalayın, kullanıcıya/isteğe anlamlı HTTP kodları (400, 403, 404, 500, vb.) döndürün.
  4. Middleware Kullanımı: Tekrarlayan doğrulama, kimlik denetimi gibi işlemler için Express middleware yaklaşımını benimseyin.
  5. Veritabanı Bağlantısı: SQL enjeksiyondan kaçınmak için parametreli sorgular (ör. db.query('... WHERE id=?', [value])) kullanın. Hassas bilgileri log’lamayın.

API Endpoint İsimlendirme

  • RESTful mantığa uyun (GET, POST, PUT, DELETE vb.).
  • Kaynak isimleri çoğul: /words, /definitions, /relations.
  • Projede tam uygulanmadıysa da plan, genelde bu yönde. Örn: “DELETE /admin/words/:id” gelecekte eklenecekse, bu formatla ilerlenebilir.

Dosya ve Klasör Yapısı

Proje şu an:

turkce-sozluk-lab/
├─ public/              # Statik dosyalar (HTML, CSS, JS)
├─ index.js             # Express app, ana sunucu
├─ package.json
├─ .env
└─ ...
  • Henüz routes, controllers gibi alt klasörler detaylı tanımlı değil. İleride genişlerse, Katkı Rehberi’nde belirtilen yapıya yakın bir ayrıştırma yapabilirsiniz.
  • Yeni eklediğiniz dosyaları mevcut yapı ve isme uygun olacak şekilde konumlandırmaya özen gösterin.

Linter ve Formatlama (Varsa)

Proje henüz ESLint/Prettier entegre etmemiş olabilir. Entegrasyon sağlamak isterseniz bir PR’da konfigürasyon dosyaları ekleyip tanıtabilirsiniz. Kod stili olarak şu basit kuralları gözetin:

  • 2 boşluk girinti,
  • Noktalı virgül (;) kullanımı tutarlı,
  • Fonksiyon parantezleri ve bloklar arası mantıklı boşluklar,
  • Gereksiz trailing whitespace’ten kaçın.

TEST VE CI/CD

Testler

  • Projede tam test altyapısı olmayabilir (ör. Jest, Mocha). Hali hazırda test dosyaları görmüyorsanız, test eklemek isterseniz proje sahibiyle veya issue açarak tartışın.
  • Geliştirme sırasında manuel testler (ör. Postman’le endpoint’leri denemek) yapılması önerilir.
  • İleride testler eklenirse, “npm test” komutuyla çalıştırıp hata alıp almadığınızı kontrol edin.

CI/CD Pipeline

  • Şu an proje tam oturmuş bir CI/CD pipeline’ına sahip olmayabilir.
  • Yerel testlerinizi yaptıktan ve çatallaşmış (fork) depo üzerinden push ettikten sonra bir PR açarak değişikliklerin geçerliliğini gösterebilirsiniz.
  • Geniş bir CI/CD planı (otomatik test, otomatik deploy vb.) gelecekte eklenebilir.

COMMIT MESAJLARI VE BRANCH YAPISI

Commit Mesajı Standardı

  • Çok katı bir kural yok, ancak düzenli commit mesajları süreç takibini kolaylaştırır. İsteğe bağlı olarak Conventional Commits tarzı (feat, fix, docs, vb.) kullanılabilir. Örn:
    • feat: add new admin endpoint for synonyms
fix: handle null pointer in word definitions
docs: update README with usage instructions
  • Kısa ve öz anlatmaya çalışın; gerekliyse açıklayıcı metni ek bir paragrafta yazın.

Branch Modeli

  • Temel yaklaşım: main dalı (veya develop varsa) + feature/..., fix/... branşları.
  • “Hotfix” gibi acil durumlar mainden çıkarılabilir.
  • Büyük projelerde develop > main merge akışı kullanılıyor olsa da bu projede ölçek gereği basit bir model yeterli.
